<task type="auto">
  <name>Task 2: Fix weekly review live updates after clarify</name>
  <files>app/web_ui/js/review.js</files>
  <action>
    The Weekly Review uses MutationObserver to watch for the clarify modal closing, then calls `_renderCurrentItem()`.

    Read `_watchClarifyClose()` and `_watchFlyoutClose()` in review.js (~line 840).

    Current approach: watches clarifyModal for losing `is-open` class. The bug is that after clarifying an inbox item, the review card doesn't update.

    Investigate:
    1. Check if `_watchClarifyClose()` is actually called when the "Process" button flows through the flyout → clarify path.
    2. Verify the MutationObserver is watching `{ attributes: true, attributeFilter: ["class"] }` on the right element.
    3. Check what `_renderCurrentItem()` does — does it re-query inbox items from live state, or from a stale snapshot captured at session start?

    The most likely root cause: `_renderCurrentItem()` uses a stale task list captured when the review session started, so even after state changes the card doesn't reflect the update.

    Fix: ensure `_renderCurrentItem()` (and the methods that build the current section's items) re-reads from `taskManager.getTasks()` live rather than from a session-time snapshot. If the observer isn't being set up in the right place, fix the setup call chain too.

    Do not change the overall MutationObserver-based approach — it's the right pattern for this.
  </action>
  <verify>

    Manual: start a Weekly Review, open an inbox item via "Process", complete the clarify modal, confirm the review card updates to reflect the task's new status.

<task type="auto">
  <name>Task 3: Fix sync toast overlapping feedback button</name>
  <files>app/web_ui/css/style.css</files>
  <action>
    Current state (style.css ~516):
    ```css
    /* Offset toasts above the feedback widget button so they don't overlap */
    body.is-admin .alerts {
      bottom: calc(var(--space-5) + 44px + var(--space-2));
    }
    ```

    The feedback widget is `position: fixed; bottom: var(--space-5); right: var(--space-5)` and is only visible for admin users. The toast container `.alerts` is also `position: fixed; bottom: var(--space-5); right: var(--space-5)`.

    Investigate why the toast is still covered:
    1. Check if `body.is-admin` is reliably set when the feedback widget is visible (ui.js ~254).
    2. Check mobile: at ≤860px, does the `.alerts` offset still apply? Is there a mobile media query that resets `.alerts { bottom }` without the admin override?
    3. Check if `body.flyout-open .feedback-widget { right: ... }` moves the widget but not the alerts container, causing misalignment.

    If the offset is only missing on mobile: add a mobile-specific rule under the existing `@media (max-width: 860px)` block:
    ```css
    body.is-admin .alerts {
      bottom: calc(12px + 44px + var(--space-2));  /* 12px = mobile bottom spacing */
    }
    ```

    If the offset calculation is wrong in general: correct the base rule. The feedback toggle is 44px height, positioned at `var(--space-5)` from the bottom. The offset should be `var(--space-5) + 44px + var(--space-2)` — verify this is correct and that `--space-5` and `--space-2` resolve to expected values.

    Fix the root cause. Do not add `!important` or z-index hacks.
  </action>
