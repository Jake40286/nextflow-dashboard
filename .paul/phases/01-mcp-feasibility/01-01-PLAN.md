<objective>
## Goal
Produce a written Decision Record that answers four questions and ends with a clear go/no-go recommendation for v1.1 Phase 1's walking-skeleton PoC:
1. Which Python MCP implementation (official `mcp` SDK, alternative SDK, or hand-rolled JSON-RPC)?
2. What sidecar shape (Docker image, port, volume access pattern) integrates cleanly with the existing `docker-compose.yml`?
3. What is the path of least resistance for the MCP server to call `PUT /state` with `If-Match: <rev>` and survive 409 conflicts?
4. What auth posture does Phase 1 ship with (and what gets deferred to Phase 3)?

## Purpose
Phase 1 is a feasibility gate. The next plan (01-02) builds a walking-skeleton PoC, but only if the answers below show MCP-in-Python is practical. A no-go answer here is *cheap* and is a valid milestone outcome — it pauses or re-scopes v1.1 instead of forcing a bad build.

This research plan also serves as the v1.1 architectural anchor: every later plan (full tool surface, security hardening) will reference the decisions made here.

## Output
A new file: `.paul/phases/01-mcp-feasibility/DECISIONS.md` containing:
- SDK survey table (≥2 candidates, pros/cons, license, last-release date, dependency footprint)
- Sidecar shape proposal (image, port, volume vs. HTTP-only access pattern)
- Sync-path strategy (how the MCP server obtains `_rev`, sends `If-Match`, retries on 409)
- Auth posture for Phase 1 (and deferred items routed to Phase 3)
- **Go/no-go recommendation** with explicit risks, blast radius, and rollback path

No production code is written in this plan.
</objective>

    Then write two sections to DECISIONS.md:

    **Sidecar Shape** — propose: base image (python:3.11-slim or 3.12-slim), exposed port (must not collide with 8002), how it accesses data: (a) bind-mount `./data` directly, OR (b) HTTP-only via the main app on the docker network. State which is recommended and why. Include a draft `docker-compose.yml` service block for the sidecar.

    **Sync Path Strategy** — explain end-to-end how the MCP server will perform a write:
    1. How it obtains the current `_rev` (GET /state from inside docker network).
    2. How it constructs the write payload (delta-style? full-state?).
    3. How it sends `If-Match: <rev>`.
    4. How it handles 409 (re-fetch state, re-apply mutation, retry — mirroring `flushRemoteQueue`'s retry loop).
    5. Retry cap and what happens on exhaustion.
    6. Identify any race with browser clients that the JS retry loop already handles, vs. any new race the MCP server introduces.

    Avoid: assuming the MCP server can write `state.json` directly. The optimistic-locking constraint is non-negotiable.
  </action>
