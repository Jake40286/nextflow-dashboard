<objective>
## Goal
Build a walking-skeleton MCP sidecar that exposes one tool, `create_task`, end-to-end. An LLM client (Claude Desktop or Claude Code) running on the same host as the docker stack must be able to call `create_task({title: "..."})`, watch a new inbox task appear at http://localhost:8002, and have the write go through the existing `PUT /state` optimistic-locking path with `If-Match`.

## Purpose
This plan is the GO verdict from 01-01 in code form. Its job is to prove three things in production:

1. The official `mcp` Python SDK works in a docker-compose sidecar talking to the existing `web` service over the docker network.
2. The 409-retry loop ported from `data.js:744-787` survives a real write race against a browser client.
3. Claude Desktop / Code on the same host can connect to the sidecar's streamable-HTTP transport and discover the `create_task` tool.

If any of these break, that's a **Code** issue (per the diagnostic-routing convention from APPLY), not a Spec issue — DECISIONS.md is the spec, and it's locked.

## Output
- A new MCP sidecar service running alongside `web` in the existing docker-compose stack, reachable only from `localhost:8003` on the docker host.
- One tool registered: `create_task(title: str, status?: str = "inbox")`.
- A new task visible in the browser at http://localhost:8002 within seconds of the tool call.
- Updated `.paul/phases/01-mcp-feasibility/01-02-SUMMARY.md` documenting actual deps installed, container size, and any drift from DECISIONS.md.

**No source files in `app/web_ui/`, `tests/`, or `app/server.py` are modified.** The main `web` service is untouched.
</objective>

<task type="auto">
  <name>Task 2: Implement mcp_server.py with create_task tool + 409-retry loop</name>
  <files>app/mcp_server.py</files>
  <action>
    Build the MCP server using the official SDK. Target ~80-120 lines. Structure:

    1. **Imports + config**
       - Import the MCP SDK's server primitives (consult `mcp==1.27.0` docs; the entry point is roughly `from mcp.server.fastmcp import FastMCP` or `from mcp.server import Server` depending on idioms — use whichever the SDK README recommends for streamable-HTTP).
       - Read `NEXTFLOW_API_URL` from env (default `http://web:8000` for the docker network case).
       - Read `MCP_HOST` (default `0.0.0.0`) and `MCP_PORT` (default `8001`) from env.

    2. **`fetch_state()` helper** — async GET `{NEXTFLOW_API_URL}/state` via `httpx.AsyncClient`. Returns the parsed JSON body (which includes `_rev`).

    3. **`write_state_with_retry(mutate_fn)` helper** — the canonical pattern from `data.js:744-787`, ported:
       - Up to `MAX_RETRIES = 3` attempts.
       - Each attempt: `state = await fetch_state()`; `new_state = mutate_fn(state)`; PUT `{NEXTFLOW_API_URL}/state` with `If-Match: <state["_rev"]>` and the full new_state body.
       - On 409: parse the response body as the new server state, do NOT trust the local copy; loop with the fresh state.
       - On 200: return the new `_rev`.
       - On retry exhaustion: raise an exception that the MCP tool surface translates to a tool error visible to the LLM caller.
       - Use a single `asyncio.Lock` (`_mcp_state_lock`) to serialize concurrent tool calls within this process — prevents two in-flight `create_task` calls from racing each other (DECISIONS.md "Race Conditions" row 3).

    4. **`create_task` tool** — register with the SDK's tool decorator. Schema:
       - `title: str` (required) — the task title shown in the inbox.
       - `status: str` (optional, default `"inbox"`) — one of nextflow's STATUS enum values (`inbox`, `next`, `doing`, `waiting`, `someday`).
       - Description (one sentence): "Create a new nextflow task. The task lands in the user's Inbox by default."
       - Implementation: build a task dict with required fields the existing `data.js` produces — `id` (uuid hex), `title`, `status`, `createdAt`, `updatedAt` (both nowIso), `_fieldTimestamps: {}`, `_source: "mcp"`. Do NOT set fields that browser-side code derives lazily (e.g. `slug`).
       - Mutator function: append the new task dict to `state["tasks"]`.
       - Return value: `{"id": <task_id>, "_rev": <new_rev>}`.

    5. **Server startup** — bind streamable-HTTP transport on `MCP_HOST:MCP_PORT`. Print one banner line on startup (`MCP server listening on ...`) so logs are useful.

    Avoid:
    - Do NOT call `app/web_ui/js/data.js`'s mutators — this is Python; we build the task dict directly. (The merge logic stays on the server side.)
    - Do NOT include any auth check (token, header) — Phase 1 PoC is localhost-only by binding, per DECISIONS.md "Auth Posture: Phase 1".
    - Do NOT add image upload, project creation, or any tool other than `create_task`. Phase 2's job.
    - Do NOT include `completionLog`, `reference`, or `completedProjects` in the PUT body — those are per `flushRemoteQueue`'s `_completionsDirty=false` path (`data.js:738-743`). Stripping them lets the server leave `completed.json` untouched.
    - Do NOT supply a `_rev` field in the PUT body — server strips it (`server.py:813`).
  </action>

- `app/web_ui/**` — no UI changes in this plan. (The new task showing up in the Inbox is via the existing `statechange` re-render path.)

## SCOPE LIMITS
- Only `create_task` is exposed. No `create_project`, no `list_tasks`, no `update_task_status`. Those are Phase 2.
- No auth headers, tokens, or shared secrets. Localhost-only bind is the entire security model for this plan.
- No internet-exposure docs or warnings. Phase 3 owns docs.
- No tests, no CI integration. Verification is manual.
- No `requirements.txt` for the main app. The `requirements-mcp.txt` file is sidecar-scoped only.
- No JS file restructure. Adjacent only; if a clean opportunity surfaces, log it as a deferred item, do not act on it here.
