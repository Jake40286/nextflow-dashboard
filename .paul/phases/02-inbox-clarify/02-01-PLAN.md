<objective>
## Goal
Fix two gaps in the clarify modal: the "New project" path doesn't pre-fill the project name from the task title and leaves no way to set project details after routing; and the description field is hidden during clarification even though the element and listener already exist.

## Purpose
The clarify flow is the highest-frequency interaction in the app. Both items break user flow — the project conversion path is a dead end (task title discarded, no project setup window), and the description field should be the natural place to add context while processing.

## Output
- `app/web_ui/js/ui.js` modified (3 functions + 1 closure)
- `.paul/phases/02-inbox-clarify/02-01-SUMMARY.md` created
</objective>

## AC-1: New project name pre-fills from task title
```gherkin
Given a task is open in the clarify modal and has a title in the editing area
```

## AC-2: Project flyout opens after new-project routing
```gherkin
Given a user has typed a project name and clicked Done to finalize routing
When the clarify modal closes after a new-project conversion
Then the project flyout opens automatically for the newly created project
And the user can set description, area, and notes without navigating away from inbox
```

```gherkin

When the modal first opens (any task, with or without existing description)

And edits are carried forward into finalizeClarifyRouting
```

    In `finalizeClarifyRouting()` (~line 6387), capture the converted project ID before

    ```js
    // Capture before _completeClarifyStep resets clarifyState
    const convertedProjectId = this.clarifyState.convertedProjectId || null;
    const inSession = !!this.processSession;

    this.taskManager.notify("info", `✓ Routed to ${dest}`, { actions });
    this._completeClarifyStep("routed");

    if (convertedProjectId && !inSession) {
      this.openProjectFlyout(convertedProjectId);
    }
    ```

    `this.taskManager.notify(...)` call — at the end of the function, after all the updates
    are computed and taskManager.updateTask has been called.

    Avoid: opening the flyout during a process session (batch clarify) — the next task
    would queue up and the two overlapping states would conflict.
  </action>
