### Intent change: convert-to-project semantics

**Captured:** 2026-05-06 during UAT of AC-1
**Status:** Not a defect against plan 02-01 — AC-1 is satisfied as written. This is an updated user intent that exceeds the original spec.

**User intent:** When converting a task to a new project, the task should *become* the project — i.e. the task is consumed (or renamed, or auto-replaced with a generated child task) rather than simply having its `projectId` reassigned. Underlying invariant the user wants to start enforcing: **a project should never exist without at least one task.**

**Why not log as UAT:** The shipped code does exactly what AC-1 prescribed (pre-fill the project name with the task title). The new behavior is additive scope, not a regression. Treating it as a UAT bug would entangle the AC-1 PASS with new work.

**Suggested handling:**
- Capture as a new feedback record in `data/feedback.json` (improvement, panel: inbox).
- Surface in ROADMAP as a new Phase 2 (or 3) plan, e.g. "02-02: convert-to-project semantics + project must-have-task invariant".
- Touches: `convertTaskToProject()` in `ui.js` (~line 8049), `finalizeNewProject` in clarify modal flow, and possibly `addProject` / project deletion guards in `data.js` to enforce the invariant.
