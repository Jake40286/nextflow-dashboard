<objective>
## Goal
Open the **Project entity** to LLM-driven creation and reading via the MCP sidecar, including the headline atomic decomposition tool. After this plan ships, an MCP-aware LLM can:

- Create a project with full GTD metadata (`create_project`)
- Discover existing projects to avoid duplicates (`list_projects`, `get_project`)
- **Atomically create a project + N child tasks in a single PUT** (`create_project_with_tasks`) — the flagship use case ("decompose 'plan a Tokyo trip' into a project plus next-action tasks")

## Purpose
Phase 1 proved the plumbing. Phase 2 fills the tool surface. This plan delivers the most-asked-for capability first: the user said the headline use case is "give a vague description of a project they want to build and let the LLM decide the steps required." `create_project_with_tasks` IS that capability.

By keeping all four new tools in this single plan, we ship a coherent vertical slice (Project entity end-to-end) rather than half-finishing Projects to start on Tasks. Plan 02-02 will mirror the structure for the Task entity.

## Output
- `app/mcp_server.py` grows from 1 tool to 5 (`create_task` retained from 01-02 + 4 new project-side tools).
- The LLM-facing tool descriptions explain GTD-specific fields (`areaOfFocus`, `theme`, `status`, `vision`, `deadline`) inline — so projects created via MCP are semantically right, not just structurally valid.
- Smoke test demonstrates: create a project; list projects; create_project_with_tasks creates one project + 3 child tasks atomically with `_rev` incrementing exactly once; the new project appears at http://localhost:8002 in the Projects panel with all child tasks under it.
- All MCP-created entities tagged with `_source: "mcp"` (consistent with 01-02's task pattern).

**No source files outside `app/mcp_server.py` are modified.** Main app, JS, tests, compose, deps — all untouched.
</objective>

<task type="auto">
  <name>Task 3: Add `create_project_with_tasks` atomic decomposition tool — flagship</name>
  <files>app/mcp_server.py</files>
  <action>
    Add a new `@mcp.tool()` async function `create_project_with_tasks` that:

    Parameters:
    - `project: dict` — same shape as `create_project`'s args, but as a single dict (`{title: str, description?: str, vision?: str, areaOfFocus?: str, theme?: str, status?: str, deadline?: str}`).
    - `tasks: list[dict]` — array of task dicts, each `{title: str (required), status?: str (default "next"), description?: str, contexts?: list[str], peopleTags?: list[str]}`.

    Behavior:
    - Validate the project (same validation as `create_project` — status enum check; non-empty title).
    - Validate `tasks` length: must be 1-50. Empty list → `ValueError("at least one task required")`. >50 → `ValueError("max 50 tasks per atomic decomposition; split into multiple calls")`. Phase 3 will refine this cap; 50 is a sane initial guard against LLM hallucinations producing 200-task lists.
    - Validate each task: non-empty title; status (if provided) is in the task-status enum.
    - Build the new project dict (same shape as `create_project`'s output) — call its id `new_project_id`.
    - Build the new task dicts — each gets a fresh uuid, `projectId: new_project_id`, `status: <given or "next">`, `_source: "mcp"`, ISO timestamps. Default task status is "next" (NOT "inbox") because tasks under a planned project are explicit next actions.
    - Mutator function: append the new project to `state["projects"]`, append ALL tasks to `state["tasks"]`. Return the modified state.
    - Use `_write_state_with_retry(mutator)` — single PUT carries the project + N tasks. The 3-retry loop from 01-02 already handles 409 contention.
    - Return `{"project_id": new_project_id, "task_ids": [...], "_rev": <new_rev>}`.

    Docstring (the LLM-facing one — write carefully):
    > "Atomically create a project and its initial tasks in a single transaction. Use this when the user describes a goal that needs to be decomposed into a project plus child tasks (e.g., 'plan a vacation', 'set up the new server'). The project's tasks default to status='next' since they are explicit next actions of a planned project. Limit: 50 tasks per call — split larger decompositions into follow-up create_task calls."

    Avoid:
    - Do NOT call `create_project` and then `create_task` N times — that's not atomic and produces N+1 `_rev` increments. The whole point of this tool is one PUT, one `_rev` bump.
    - Do NOT set task `status: "inbox"` for tasks created here — the user has already clarified the project, so child tasks are next actions, not unprocessed input. (If the LLM needs to add inbox-bound tasks later, it uses `create_task` with explicit `status="inbox"`.)
    - Do NOT add tasks to projects that already exist — that's a different operation (Phase 2's task tools will offer `create_task(projectId=...)` later). This tool is exclusively for *new* project + *new* tasks together.
  </action>
