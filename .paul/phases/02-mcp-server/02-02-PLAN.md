<objective>
## Goal
Open the **Task entity** to LLM-driven discovery and follow-up via the MCP sidecar. After this plan ships, an MCP-aware LLM can:

- Find existing tasks (`list_tasks` with filters; `get_task`)
- Mark tasks done / change task status (`update_task_status`)
- Re-link a task to a different project (`set_task_project`)
- Append context notes to a task (`add_task_note`)

This completes the read/light-update half of Phase 2's tool surface, complementing 02-01's project tools + atomic decomposition.

## Purpose
Round-trip ergonomics. With 02-01 the LLM can create projects + tasks; with 02-02 the LLM can answer "what's on my pending list?", "mark the dentist task as done", "move this task to the Tokyo trip project", and "add a note that I tried calling them at 3pm". These are the natural follow-ups to creation flows.

`update_task_status` carries the most weight: it's the LLM-facing path for "mark X as done" (move to status `"completed"` — actually status moves through the task lifecycle and final completion goes through `completionLog`; this plan needs to read `data.js`'s `completeTask` flow before deciding how to expose "done").

## Output
- `app/mcp_server.py` grows from 5 tools to **10**: previous + `list_tasks`, `get_task`, `update_task_status`, `set_task_project`, `add_task_note`.
- Tool docstrings document task field semantics: `status`, `contexts`, `peopleTags`, `areaOfFocus`, `dueDate`, `followUpDate`, `effortLevel`, `timeRequired` — same fidelity as 02-01's project schemas.
- All updates write `_fieldTimestamps` correctly (per `MERGE_FIELD_GROUPS` in `data.js`) so concurrent browser writes don't silently overwrite MCP-driven changes.
- Smoke test verifies a round trip: create a task via existing `create_task`, list it, get it, update its status, set its project, append a note — each step's effect visible in the browser.

**No source files outside `app/mcp_server.py` are modified.** Main app, JS, tests, compose, deps — all untouched.
</objective>

## AC-1: `list_tasks` filters work and the response is LLM-friendly
```gherkin
Given the user has tasks across multiple statuses, projects, and contexts
When the LLM calls `list_tasks(status="next")`
Then the response is an array of task summaries, one per active task with `status == "next"`
And each entry has `id`, `title`, `status`, `projectId`, `contexts`, `peopleTags`, `areaOfFocus`, `dueDate`, `followUpDate`
And internal sync fields (`_fieldTimestamps`, `_source`, `slug`, `originDevice*`) are stripped from the response

When the LLM calls `list_tasks(project_id="<id>")`
Then only tasks with `projectId == <id>` are returned (any status)

When the LLM calls `list_tasks(context="@Phone")`
Then only tasks where `"@Phone" in task.contexts` are returned

When filters are combined (e.g. `list_tasks(status="next", project_id="<id>")`)
Then results match ALL filters (logical AND)

When `list_tasks()` is called with no filters
Then the default returns all active tasks (those still in `state.tasks` — completed tasks live in `completionLog` and are not returned by this tool)
```

## AC-2: `get_task` returns full task with notes
```gherkin
Given a task with id `<id>` and one or more notes
When the LLM calls `get_task(id=<id>)`
Then the response includes the full task record (excluding internal sync fields)
And `notes` is an array with the same items the UI shows
And calling `get_task(id="bogus")` raises a tool error visible to the LLM
```

## AC-3: `update_task_status` mutates correctly with `_fieldTimestamps` bump
```gherkin
Given a task currently in `status: "next"`
When the LLM calls `update_task_status(id=<id>, status="doing")`
Then the tool returns success with the new `_rev`
And the task's `status` is updated to "doing" on disk
And `task._fieldTimestamps.status` is set to a new ISO timestamp (post-update)
And `_rev` advances by exactly one
And the task's `updatedAt` is bumped

When the LLM calls `update_task_status(id=<id>, status="bogus")`
Then a `ValueError` is raised; no write occurs; `_rev` does not advance
```

## AC-4: `set_task_project` re-links a task
```gherkin
Given a task currently linked to `projectId: "project-A"`
When the LLM calls `set_task_project(id=<task-id>, project_id="project-B")`
Then `task.projectId` is updated to `"project-B"` on disk
And both projects' UIs (browser refresh) reflect the move

When the LLM calls `set_task_project(id=<task-id>, project_id=None)` or `project_id=""`
Then the task is unlinked (projectId set to null)

When the LLM calls `set_task_project(id="bogus", project_id="<any>")`
Then a `ValueError` is raised
```

```gherkin
And the existing 2 notes are unchanged
And `_rev` advances by exactly one

When the LLM calls `add_task_note(id=<id>, text="")` or whitespace-only
Then a `ValueError` is raised; no write occurs
```
