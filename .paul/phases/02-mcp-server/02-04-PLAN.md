<objective>
## Goal
Stand up a pytest-based test suite for the `app/mcp_server/` package that locks in the codified invariants from 02-01/02/03 (per-group LWW timestamp bumps, `task-<uuid>` ID format, `_rev` Δ = +1 atomicity, append-only note semantics, 409-retry path, sidecar handshake / starlette resolution).

## Purpose
The 02-03 refactor and three deferred-cleanup landings shipped untested beyond a single live smoke. STATE flags `starlette<2` resolution as a known drift risk and 02-02's per-group LWW pattern as an easy-to-regress invariant. A fast unit suite + one live integration test gives us a regression net before Phase 3 starts adding `complete_task` and arbitrary field updates on top of the same plumbing.

## Output
- `tests/mcp_server/` package mirroring the `app/mcp_server/` DAG (sync, state_helpers, tools/projects, tools/tasks, integration).
- `requirements-mcp-dev.txt` pinning `pytest`, `pytest-asyncio`, `pytest-httpx` (sidecar-dev-scoped; not loaded into the production image).
- All unit tests green via `pytest tests/mcp_server/ --ignore=tests/mcp_server/test_integration.py`.
- Integration test green when run against a live `docker compose up` stack via `pytest tests/mcp_server/test_integration.py` with the `--integration` opt-in flag (skipped by default).
</objective>

## AC-1: 409 retry path is covered end-to-end
```gherkin
Given a mocked nextflow `web` service that returns 409 on first PUT and 200 on second
When `_write_state_with_retry(mutate_fn)` is called
Then the mutator runs twice (once per attempt), the second PUT carries the updated `If-Match` rev from the 409 response body, and the helper returns the 200 response body
And given the mock returns 409 on every attempt up to MAX_RETRIES
When the helper is called
Then it raises `RuntimeError` after exactly MAX_RETRIES attempts
```

## AC-2: Optimistic-locking wire contract is asserted
```gherkin
Given a mocked `web` returning a state with `_rev: 42`
When any write tool runs (`create_task`, `create_project`, `create_project_with_tasks`, `update_task_status`, `set_task_project`, `add_task_note`)
Then the outbound PUT to `/state` carries header `If-Match: 42`
And the PUT body has `_rev`, `completionLog`, `reference`, `completedProjects`, and `projectActivityLog` stripped
```

## AC-3: Codified invariants are locked in (per-group LWW, ID format, atomicity, append-only notes)
```gherkin
Given any task created by `_build_task_record` (directly or via `create_project_with_tasks`)
Then its `id` matches the regex `^task-[0-9a-f-]{36}$` (matches data.js generateId('task'))

Given `update_task_status(task_id, "next")` is invoked
Then the resulting task has `_fieldTimestamps["status"]` set to a fresh ISO timestamp (per-group LWW bump from 02-02 Decision)
And `set_task_project` does NOT bump `_fieldTimestamps` (projectId is whole-task LWW per 02-02 Decision)

Given `create_project_with_tasks(project, [t1, t2, t3])` runs against a mocked `web`
Then exactly ONE PUT /state is issued (not four), proving the `_rev` Δ = +1 atomicity guarantee from 02-01

Given a task with two pre-existing notes
When `add_task_note(task_id, "third note")` runs
Then the resulting task.notes is length 3 with the new note APPENDED (existing notes preserved verbatim, ordering stable, `id` matches `^note-[0-9a-f-]{36}$`)
```

## AC-4: Live integration test proves end-to-end with real `web`
```gherkin
Given `docker compose up -d` is running and the user invokes pytest with `--integration`
When the test connects to the live sidecar at `http://127.0.0.1:8003/mcp` and calls `create_task(title="02-04 integration smoke")`
Then the call returns success, a follow-up `GET /state` against `web` shows the new task with `_source: "mcp"` and `id` matching `^task-[0-9a-f-]{36}$`
And the test then deletes its smoke task by tombstoning to leave state clean
And the test fails fast with a clear "compose not running" message if the sidecar is unreachable, instead of hanging
```

## AC-5: Suite is fast and CI-friendly by default
```gherkin
Given a developer runs `pytest tests/mcp_server/`
Then unit tests complete in under 5 seconds with zero network calls
And the integration test is automatically skipped (no compose dependency) unless `--integration` is passed
```
