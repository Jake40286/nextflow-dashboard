<objective>
## Goal
Add the data layer for the project activity / change log feature: a new `projectActivityLog` collection that lives in `completed.json`, mirrored on the wire and merged on the server using the same accumulator pattern as `completionLog`. Wire `_logActivity()` emission into all 8 MVP mutation sites so every status change, project assignment, task lifecycle event, and project lifecycle event produces a structured, human-readable entry. No UI in this plan â€” plan `04-02` will surface the log in the project flyout.

## Purpose
Resolves feedback `7868b077` (project activity / change log). Per `.paul/phases/04-projects-features/DISCOVERY.md` (HIGH confidence), the codebase already has the right primitive â€” the `completionLog` split-persistence + server-merge + lazy-load pattern. This plan adds an analogous collection without inventing new persistence machinery, and instruments emission so the data is collected from day one. Splitting "data" from "UI" lets us deploy the data layer, watch it work in production for normal use, and then build the UI in `04-02` with confidence the underlying log is correct.

## Output
- New `state.projectActivityLog` array initialised to `[]` in default state shape
- New `_logActivity(type, projectId, details)` helper in `data.js` that pushes a structured entry and marks `_completionsDirty`
- New `getProjectActivity(projectId)` accessor (sorted oldestâ†’newest, returns array)
- Server merges incoming `projectActivityLog` into `completed.json` like `completionLog` (no replacement)
- 8 emission sites instrumented:

  - **Project events:** `addProject`, `updateProject` (status change), `completeProject`
- Tests covering: emission per event type, server-merge accumulator behavior, accessor sort order, `_completionsDirty` set on emit

## AC-1: Activity entries are emitted for the 8 MVP event types

```gherkin
Given a task is assigned to a project
When the user mutates the task or project in any of these ways:
  - changes the task's status (next â†’ doing, etc.)
  - moves the task into / out of / between projects
  - completes the task
  - deletes the task
  - restores a completed task
  - creates a project (with or without an initial task)
  - changes a project's status (Active â†” OnHold â†” Completed)
  - completes a project
Then exactly one new entry is appended to state.projectActivityLog for that mutation
  And the entry has shape { id, projectId, type, taskId?, taskTitle?, actor, ts, before, after }
  And `_completionsDirty` is set to true
  And task moves between projects produce TWO entries (one for the source project with after=newId, one for the destination project with before=oldId)
```

## AC-2: Server merges projectActivityLog accumulatively

```gherkin
Given the server has 5 entries in completed.json projectActivityLog
  And a client sends a PUT with 3 entries (1 overlapping by id, 2 new)
When the server processes the PUT
Then completed.json's projectActivityLog contains 7 entries (5 + 2 new; the overlapping id keeps the newer ts version)
  And no entries are dropped from the server's existing log even if the client's payload omits them
  And state.json is unaffected
```

## AC-3: getProjectActivity accessor returns scoped sorted entries

```gherkin
Given state.projectActivityLog contains entries for multiple projects
When the user calls taskManager.getProjectActivity("proj_abc")
Then only entries whose projectId === "proj_abc" are returned
  And entries are sorted by ts ascending (oldest first)
  And the original state.projectActivityLog is not mutated
  And calling with an unknown projectId returns an empty array
```

## AC-4: Sync payload is conditional and the field round-trips correctly

```gherkin
Given the user mutates a tracked field (e.g., changes a task's status)
When the client flushes the next PUT
Then the payload includes projectActivityLog
  And after the PUT succeeds, _completionsDirty is cleared
  And subsequent unrelated edits do NOT include projectActivityLog in their PUTs (per existing _completionsDirty behavior)
  And on next page load, state.projectActivityLog is hydrated from /completed
```

## SCOPE LIMITS
- **No UI in this plan.** No project-flyout tab, no rendering, no DOM, no CSS. Plan 04-02 owns all of that.
- **No backfill.** Existing projects start with empty activity logs. Synthetic "imported from history" entries are explicitly excluded per the discovery's caveats.
- **MVP event types only.** No emission for note edits, list-item edits, dueDate, contexts, area-of-focus changes, or title renames. If they come up in the planning conversation, capture as deferred issues â€” do not add to this plan.
- **No server-side prune yet.** The log accumulates indefinitely. Prune policy can be added in a future plan if storage actually becomes a problem.
- **No `task.completed` writing to BOTH completionLog AND projectActivityLog as a refactor concern.** The plan keeps both writes (completionLog write already happens; projectActivityLog write is new and additive). That's intentional duplication â€” different consumers.
