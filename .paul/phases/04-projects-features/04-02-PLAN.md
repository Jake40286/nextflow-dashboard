<objective>
## Goal
Make the project activity log visible. Add an "Activity" section to the project flyout that lists the recent events for the current project, ordered newest-first, with each event rendered in plain English (e.g., "Marked 'Email vendor' complete", "Moved 'Buy chips' from inbox to next"). Lazy-load the underlying data on the first flyout open per session.

## Purpose
Closes feedback `7868b077` — the user wanted "a detailed log of what changed on tasks who are assigned to a project." Plan 04-01 collected the data; this plan surfaces it. The activity section sits inside the existing project flyout (next to outcome / notes / tasks) so users see history in the same place they manage the project — no mode-switching, no separate panel.

## Output
- New "Activity" section appended to the project flyout content, below the task list
- Each entry rendered as one row: icon/marker + plain-English event + relative timestamp + actor (subdued)
- Empty-state copy when a project has no activity yet ("No recorded activity for this project yet.")
- Lazy-load: first flyout open per session triggers `ensureCompletedLoaded()`, then re-renders the section once entries are in memory
- Sort: newest first (the underlying `getProjectActivity` returns oldest first, so the renderer reverses for display)
- CSS that matches the existing flyout sections (outcome, notes) — same indentation, label style, muted treatment
</objective>

### Where the activity section gets rendered
- `app/web_ui/js/ui.js:1973` — `renderProjectFlyout(project)` is the function that builds the flyout content
- `app/web_ui/js/ui.js:~2042` — `content.innerHTML = ""` clears the content area at the start of each render
- The existing scroll order is: outcome → notes (collapsible) → task list. The activity section will go AFTER the task list, at the bottom of the content scroll, matching how the eye reads the project (what's it for → my notes → tasks → history)

### Existing lazy-load pattern (to mirror)
- `app/web_ui/js/ui.js:828` — Statistics/Reports panels call `this.taskManager.ensureCompletedLoaded().then(() => { ... re-render ... })`. Same pattern works here: kick off the load when the flyout opens, re-render when done.
- `data.js` `ensureCompletedLoaded` is idempotent — second-and-later calls in a session are no-ops. Safe to call on every flyout open.

### Data hooks already in place (from 04-01)
- `taskManager.getProjectActivity(projectId)` → returns scoped array, oldest first
- Entry shape: `{ id, type, projectId, taskId, taskTitle, actor, ts, before, after }`
- 8 event types: `task.statusChange`, `task.projectAssign`, `task.completed`, `task.deleted`, `task.restored`, `project.created`, `project.statusChange`, `project.completed`

## AC-1: Activity section renders for any project's flyout

```gherkin
Given a project exists
  And the user opens that project's flyout
When the flyout content has finished rendering
Then an "Activity" section appears at the bottom of the content
  And the section has a heading "Activity" matching the visual style of other section labels in the flyout
  And if the project has activity entries, they are listed newest first
  And each row shows a short icon/marker, plain-English text, a relative time (e.g., "2h ago"), and the actor as muted text
  And if the project has NO entries, the section shows the empty-state copy "No recorded activity for this project yet."
```

## AC-2: Plain-English rendering for all 8 event types

```gherkin
Given a project's activity log contains entries of each MVP event type
When the activity section renders those entries
Then each renders as a single readable line, with the task title (when present) quoted, and the actor name shown subtly to the right or in a tooltip:
  - task.statusChange      → 'Moved "<title>" from <before> to <after>'
  - task.projectAssign     → 'Added "<title>"' (before=null), 'Removed "<title>"' (after=null), 'Moved "<title>" in/out of project' (both set)
  - task.completed         → 'Completed "<title>"'
  - task.deleted           → 'Deleted "<title>"'
  - task.restored          → 'Restored "<title>"'
  - project.created        → 'Project created'
  - project.statusChange   → 'Status changed from <before> to <after>'
  - project.completed      → 'Project marked complete'
And unknown event types render a safe fallback ('Recorded activity').
```

## AC-3: Lazy-load on first flyout open per session

```gherkin
Given the user has not opened any project's flyout this session
  And the activity log is on disk in completed.json but not yet hydrated into state
When the user opens a project flyout for the first time
Then the flyout opens immediately with the activity section showing a placeholder ("Loading activity…")
  And taskManager.ensureCompletedLoaded() is called in the background
  And once that completes, the activity section re-renders with the real entries
  And subsequent flyout opens in the same session do NOT trigger another /completed fetch (ensureCompletedLoaded short-circuits)
```

## AC-4: Visual fit with the rest of the flyout

```gherkin
Given the activity section is visible in the flyout
When the user looks at the layout
Then the section's heading matches the style of other flyout section labels (outcome, notes)
  And rows align to the same indentation as the task list
  And theme cycling (light/dark/custom) keeps every entry readable
  And the section scrolls with the rest of the flyout content (no nested scroll area)
```
