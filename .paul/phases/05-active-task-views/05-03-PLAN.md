<objective>
## Goal
Replace the immediate-apply multi-edit bar with a draft + Apply/Cancel pattern, and add Contexts as a tri-state multi-value field. Resolves the third sub-item of feedback `fb700fcc`.

## Purpose
The current bar applies each select's value the moment it changes and then clears the selection. The user explicitly wants to stage several field edits and commit them together. This plan also adds the multi-value field that 05-01 deferred (Contexts) and brings the bar in line with established bulk-edit patterns (Gmail/Linear/Notion-style tri-state for multi-value attributes; "Mixed" indicator for single-value fields with heterogeneous selections).

## Output
- New Apply + Cancel buttons in the multi-edit bar
- Internal `pendingBulkEdits` draft state covering all fields
- Contexts tri-state chip group with mixed-state initial render driven by the current selection
- "Mixed" placeholder behaviour on single-value selects when the selection has heterogeneous values for a field
- Selection preserved after Apply (with reconciliation against current view); draft cleared
</objective>

<acceptance_criteria>

## AC-1: Draft + Apply/Cancel for all single-value fields
```gherkin
Given I have multiple tasks selected and the multi-edit bar is visible
When I change the Status, Project, Area, Effort, or Time Required selects
Then no task is modified yet
And the selected value is visible in the select (it is NOT cleared)
And the multi-edit bar shows a count of staged changes (e.g., "3 changes pending")
And Apply + Cancel buttons appear or become enabled

When I click Apply
Then every selected task is updated with all staged values in a single batch
And a toast confirms "Updated N tasks"
And the draft state resets (selects clear, "0 changes pending")
And the selection is preserved minus any tasks that left the current view as a result of the change

When I click Cancel (or press Escape on a non-empty draft)
Then no task is modified
And the draft state resets
And the selection is preserved
```

## AC-2: Contexts tri-state chip group
```gherkin
Given I have multiple tasks selected
When the multi-edit bar renders the Contexts chip group
Then each available context appears as a chip whose initial visual state reflects the selection:
  - "All have it"  → solid filled chip
  - "Some have it" → indeterminate / dashed chip
  - "None have it" → empty / unfilled chip
And no chip starts in an "intent" state — initial render is observation only

When I click a chip
Then it cycles through user intents: no-change → add → remove → no-change
And the visual unambiguously distinguishes "add" (e.g., green ✓ overlay) from "remove" (e.g., red − overlay) from "no change" (the observed selection state)
And the "changes pending" count includes any chip not in no-change

When I click Apply
Then for each selected task: every "add" context is appended (no duplicates), every "remove" context is stripped
And tasks left untouched on Contexts are not modified for that field
```

## AC-3: Mixed-state placeholder on single-value selects
```gherkin
Given I select tasks that have different values for a given field (e.g., two have status=NEXT, one has status=DOING)
When the multi-edit bar renders
Then that select shows a "(Mixed)" placeholder option as its initial value
And the user choosing a real value stages a write that overwrites all selected tasks on Apply
And the user leaving it on "(Mixed)" stages no change for that field
```

## AC-4: Selection survives Apply; resets only on Cancel(selection) or Escape(no-draft) or explicit Clear
```gherkin
Given selection is non-empty and draft is non-empty
When I click Apply
Then selection survives, minus tasks no longer matching the current view filters

Given selection is non-empty and draft is empty
When I press Escape or click the existing ✕ Clear button
Then selection clears (matches current behavior)

Given selection is non-empty and draft is non-empty
When I press Escape
Then the DRAFT clears first (one Escape → cancel staged edits); the selection survives
And a SECOND Escape with empty draft clears the selection
```

    HTML (`index.html` around line 1446-1465):
      - Add `<button id="multiEditApply">Apply</button>` and
        `<button id="multiEditCancel">Cancel</button>` to the bar.
      - Add `<span id="multiEditDraftCount">` for "N changes pending" indicator.
      - Keep existing ✕ Clear button — it still clears the selection.

    State (`ui.js` constructor near line 230):
      - Add `this.pendingBulkEdits = {}` (empty object means no staged edits).
      - Add `this.bulkEditMixedFields = new Set()` (computed each render).

    `updateMultiEditBar()` (line 3909):
      - For each single-value select (Status, Project, Area, Effort, Time):
        a. Compute the set of distinct values across selected tasks.
        b. If size > 1: insert/select a `(Mixed)` option as the placeholder
           and record the field in `bulkEditMixedFields`.
        c. If size === 1: render the common value as the select's value,
           but DO NOT treat that as a staged change.
      - Render `multiEditDraftCount` text from `Object.keys(pendingBulkEdits).length`
        plus the Contexts chip-group intent count (see Task 2).
      - Apply button is enabled iff at least one staged change exists.

    Replace immediate-apply handlers (`setupMultiEditBar()` line 4020-4053):
      - Each select's `change` listener now writes the chosen value into
        `this.pendingBulkEdits[field]` (or deletes the key if user
        re-selects the placeholder/Mixed). It does NOT call `applyBulkField`
        and does NOT clear the select.
      - Special case for "__none__": store `null` in `pendingBulkEdits[field]`
        so Apply can distinguish "set to none" from "no change."

    New `applyBulkEdits()` method:
      - For each `(field, value)` in `pendingBulkEdits`, build a single
        `updateTask(taskId, partial)` call per task that merges all staged
        fields into one partial — instead of one `updateTask` per (task, field),
        because per-field calls produce N×M syncs and N×M activity-log entries
        when one is sufficient.
      - Then call into the new Contexts hook (Task 2) to apply add/remove
        intents in the same partial.
      - Toast: "Updated N tasks."
      - Reset draft state: `pendingBulkEdits = {}`; clear chip intents.
      - **Reconcile selection against current view:** rebuild the visible
        task ids for the active panel and remove any selectedTaskIds not in
        that set. Then re-render the bar.

    New `cancelBulkEdits()` method:
      - Reset `pendingBulkEdits = {}`; clear chip intents.
      - Re-render the bar (selects revert to their observed-state values;
        draft count → 0).
      - Selection unchanged.

    Wire Apply, Cancel, and a revised Escape handler:
      - Apply click → `applyBulkEdits()`
      - Cancel click → `cancelBulkEdits()`
      - Escape: if draft non-empty → cancelBulkEdits(); else if selection
        non-empty → clearSelection(); else no-op.

    Delete or repurpose `applyBulkField`:
      - It's only called from the old immediate-apply handlers. Once those are
        removed, this method is dead. Remove it rather than leaving as deadwood.

    CSS: minor — Apply button uses primary style; Cancel uses light style;
    "N changes pending" muted small-text. Reuse existing `.btn` classes;
    no new design tokens.

    Avoid:
      - Adding a separate "save and continue" / "save and clear" affordance —
        keeps surface area small; user can hit Clear ✕ if they want to start
        a new selection.
      - Calling `taskManager.notify()` per field — one toast per Apply.
      - Touching `selectedTaskIds` set semantics elsewhere (row click, header
        select-all, etc.) — those continue to work as before.
  </action>

       preserved (or reduced if status change moved them out of view); draft

    3. Manual: stage one change, hit Escape — staged change gone, selection

    4. Manual: select 3 tasks where 2 have different statuses. Status select

          - is-staged-remove overlay: small red "−" badge top-right
        These can reuse the existing `.settings-area-chip.is-assigned` styling
        as a starting point (see `panels/settings.js:710-723` and the
        corresponding CSS), extended with the staged-intent overlays.

    Click handler (delegate at the chip group container):
      - On chip click: cycle the intent for that context value.
        - no entry → "add"

        - For each ("remove", contextValue): filter out.
        - If the resulting array differs from the original, include

      - Don't fire updateTask if the array would be unchanged for that task.

    Cancel/reset hook: clear `pendingContextIntents`.

    Avoid:

      - Storing tag-prefix-stripped values; keep canonical "@Home" form
